VanillaKit

A lightweight, framework-less starter kit — vanilla JS, no node_modules bloat.

Why VanillaKit?

The front-end ecosystem is full of massive frameworks and fast-moving libraries. Modern JavaScript, CSS and HTML have advanced a lot in recent years, which makes many framework conveniences possible without a heavy runtime. VanillaKit embraces that: keep what’s useful (bundling, minification, live reload) and ditch what’s not (framework lock-in, bloated dependency trees).

Key Features

  • Framework-free — pure HTML, LESS, and vanilla JavaScript
  • Two modes — production (minified) & debug (unminified)
  • Optional hot reload — works with a plain static server (no special dev server required)
  • Minimal dependencies — lessc, clean-css-cli, esbuild
  • Cross-platform — Windows, macOS, Linux
  • Single HTML source — debug/prod paths injected at build time
  • Simple deployment — copy /dist to your server and you’re done

Project Structure

src/
  assets/         # images, fonts, etc.
  components/     # HTML partials / UI fragments
  js/             # application JavaScript (includes optional livereload.js)
  styles/         # LESS styles (SMACSS-inspired)
  index.html      # template with placeholders

dist/             # build output

Installation

Install the minimal build tools via the included script:

sh install-deps.sh

This installs the CLI tools used by the build: lessc, clean-css-cli, and esbuild.

Build

Production build (default)

sh build.sh

Debug build (development)

sh build.sh debug

Hot Reload (Development)

Start file-watch + live-reload:

sh watch.sh

Then serve the dist folder:

cd dist
python -m http.server 3000

Open http://localhost:3000

Philosophy

Frameworks were essential when browsers lacked features. Today, ES modules, modern CSS, async/await, and other language improvements cover many needs. VanillaKit favors small, explicit build steps and minimal tooling — full control without hidden magic.

Why LESS?

LESS is lightweight and adds useful features (variables, nesting, mixins) without forcing heavy conventions. It helps keep CSS maintainable without introducing unnecessary complexity.

CSS Structure

The project follows a SMACSS-inspired layout (Base, Layout, Modules, State, Theme). We really like SMACSS — it solves many CSS organization problems without being overly rigid. Try it; and if you don’t like it, toss the folders and do your own thing — VanillaKit is flexible.

Plugin System (Planned)

An optional plugin system based on plain ES modules is planned. Plugins will follow a simple interface (for example, exporting an init() function) so users can drop-in preprocessors, linters, or other extensions without extra build steps.

Testing (Planned)

Upcoming plans include unit tests, UI/integration tests, and documentation on how to run and contribute tests. This will help improve reliability and onboarding for contributors.

Deployment

Deploying is trivial: simply copy the contents of the /dist folder to your web server — no additional setup or npm install required. The app is ready to run immediately.
